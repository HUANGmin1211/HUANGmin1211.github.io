Java基础第七章，异常

Posted on 2019-01-12 | In Java

原创，转载请注明出处

异常和错误

异常：Expection，通常发生可以有针对性的处理方式的。

int [] a = new int [3]; 
System.out.println(a[4]);

错误：Error，通常发生后不会有针对性的处理方式。
int [] a = new int [10241024100]; //jvm的堆内存不够了

Error的发生往往都是系统级别的问题，都是jvm所在系统发生的并反馈给jvm的，无法针对处理，只能修正代码


常用异常

  • RuntimeException； 运行时异常，那些可能在Java虚拟机正常运行期间抛出的异常的超类
  • ArrayIndexOutOfBoundsException；数组角标越界异常
  • IllegalArgumentException；向方法传递了不合法或不正确的参数
  • NullPointerException；空指针异常，对象是null
  • ClassCastException；试图将对象强制转换为不是实例的子类时抛出的异常

异常自定义&体系

想要自定义符合自己程序要求的异常名称

之前的几个异常都是java通过类进行的描述，并将问题封装成对象。这是符合面向对象思想的。其实异常就是把问题封装成了对象，所以我们可以对自己所需要的问题进行类的描述

Class DemoException extends Exception    {
    DemoException(String message)    {    
        super(message);    
    }    
}

通过查看JavaAPI，看以上几个常用的异常是怎么继承的，看源码是怎么写的

Throwable类中的方法

getMessage()：获取异常信息，返回字符串。
toString()：获取异常类名和异常信息，返回字符串。
printStackTrace()：获取异常类名和异常信息，以及异常出现在程序中的位置。返回值void
printStackTrace(PrintStream s)：通常用该方法将异常内容保存在日志文件中，以便查阅。


编译时异常&运行时异常

区别

  • 编译时异常：编译器会检测的异常
  • 运行时异常：编译器不会检测的异常，不需要声明。声明也可以，无外乎就是让调用者给出处理方式

  • 编译时异常

    如果在函数内抛出Exception，编译失败，因为编译器在检查语法时发生了错误。改程序已经出现了问题，Java认为这个程序本身存在隐患，需要捕获或者声明出来（要么把问题处理掉，要么把问题标识出来让调用者知道）

声明是为了让调用者进行处理，处理完了程序继续运行，不会停止

小敏理解：在编写代码时预知到了某一部分程序（try块）可能会出现异常，并提供预先处理方法（catch块，像日常中的灭火器），保证程序的【健壮性】

运行时异常

为什么抛出RuntimeException不需要捕获不需要声明？不是功能本身发生的异常，而是因为比如调用者传递参数错误而导致功能运行失败。这时也是问题，需要通过异常来体现，但是这个异常不要声明出来的。

不声明是为了不让调用者进行处理，为了让程序停止，让调用者看到现象，并进行代码的修正。

构造函数抛出的异常到底继承Exception还是RuntimeException呢？继承Exception，就必须要写throws声明，一声明就告知调用者进行捕获，一旦问题处理了，调用者的程序会继续执行。但是如果使用到了构造函数初始化的对象的数据，那么之后的数据就都是错的，因为初始化错了。如果继承RuntimeException，不需要throws声明，这时调用是不可能编写捕获代码的，因为调用者根本不知道有问题。一旦发生了异常，调用者程序会停掉，并由jvm将信息显示到屏幕，让调用者看到问题并修正代码

捕获&声明

声明

将问题标识出来，报告给调用者

如果函数内通过throw抛出了编译时异常，而没捕获，必须通过throws进行声明，让调用者去处理

只要方法里面存在没catch掉的异常，或者catch里面抛出了新的异常但没有继续catch掉，都需要在方法后面声明异常

捕获

Java中对异常有针对性的语句进行捕获

try{
    //需要被检测的语句 
} catch（异常类 变量）{
    //异常的处理语句 
} finally{
    //一定会被执行的语句 
}

对于声明多个异常的方法，在处理时，需要定义多个catch与之对应，catch中的变量也可以多态，但是父类必须放下面。因为如果父类catch(Exception e )放在了最上面，一次性捕获了所有子类异常，那下面的catch全不执行了。只有上面的catch没捕获完，下面的catch才继续执行。且捕获父类异常没办法针对性处理

public void run() throws OneException,TwoException{
}

throw和throws的区别

  • throw用在函数内；抛出的异常的对象
  • throws用在函数上；用于进行异常类的声明，后面的异常类可以有多个，用逗号隔开

finally

finally代码块中存放的代码都是一定会被执行的，catch中有return的话，finally在return之前执行；但是如果catch和finally都有return的话，finally还是在catch中的return之前执行，但是finally的return会结束程序，catch的return不会执行

注意：有一种情况，finally也不执行。System.exit(0);//退出jvm

应用场景：只要程序中使用到了具体的资源（数据库链接、IO资源、网络连接socket等）需要释放，都必须定义在finally中。你在定义程序，不管问题发生与否，指定程序都需要执行时，就定义在finally中

try catch finally组合

try catch，异常捕获处理：对代码进行异常检测，并对检测的异常传递给catch处理

try finally：对代码进行异常检测，检测到异常后因为没有catch，一样会被jvm抛出。异常是没有捕获处理的，但是功能所开启的资源需要关闭。这种情况异常没有处理，所以方法后面需要用 throws 声明异常

try catch finally：检测异常，并传递给catch处理，并定义资源释放

try catch1 catch2 —：对抛出的异常进行针对性处理

异常在继承或实现中的使用细节

子类在覆盖父类的方法时，如果父类的方法声明异常，子类只能声明父类异常或其子类，或者不声明

当父类方法声明多个异常时，子类覆盖时，只能声明多个异常的子集

当被覆盖的方法没有异常声明时，子类覆盖时是无法声明异常的,只能转换成RuntimeException

有一个比较特别的是RuntimeException。因为它可声明可不声明，所以子类声明它也相当于没声明。因此父类没有声明RuntimeException时，子类也是可以声明的

综上：父类已经有问题了，子类不能比父类的问题还多。

若接口中没有声明异常，而实现的子类覆盖方法时发生了异常，那么无法继续throws声明，只能catch捕获。万一问题解决不了，catch中继续throws抛出，但是只能将异常转换成RuntimeException子类抛出
